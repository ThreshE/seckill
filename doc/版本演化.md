## V0 — 无锁直扣 MySQL（演示超卖）

### 版本说明

最原始的实现。不做任何并发控制，直接 `SELECT → 判断 → UPDATE` 操作数据库。

### 代码位置

| 文件 | 关键代码 |
|---|---|
| `StockService.java` | `selectById(goodsId)` → `if(stock>0)` → `setStock(stock-1)` → `updateById(stock)` |
| `PaymentService.java` | `selectOne(userId)` → `if(balance>amount)` → `setBalance(balance-amount)` → `updateById` |
| `SeckillController.java` | 三次 Feign 调用：stock → payment → order，无事务 |

### 复现方式

```sql
-- 库存设 20，50 并发压测
UPDATE goods_stock SET stock = 20 WHERE id = 1;
```

Apifox 并发测试：50 线程同时请求 `POST /api/seckill/buy?userId=10001&goodsId=1`

### 观察到的现象

```
初始库存: 20
成功订单: 25 笔      ← 超卖了！
剩余库存: 17         ← 只有 3 次更新真正生效
```

### 根因分析

**非原子的"读-改-写"三步骤**：

```java
// StockService.java 第21~38行
GoodsStock stock = stockMapper.selectById(goodsId);    // Step1: 读
if (stock.getStock() <= 0) { return fail(...); }        // Step2: 判断
stock.setStock(stock.getStock() - 1);                   // Step3: 减
stockMapper.updateById(stock);                          // Step4: 写
```

高并发下多个线程**同时**读到了相同的库存值：

```
时间线   线程A                    线程B                    DB stock
 │       SELECT stock=20                               20
 │                             SELECT stock=20         20
 │       if(20>0)=true         if(20>0)=true
 │       UPDATE stock=19                               19
 │                             UPDATE stock=19         19 ← B覆盖了A！
 │       return success        return success
 │       创建订单#1             创建订单#2
```

**两个问题叠加：**

| 问题 | 表现 | 原因 |
|---|---|---|
| **超卖** | 20 库存卖出 25 单 | 多个线程同时通过 `if(stock>0)` 检查 |
| **更新丢失** | 25 次扣减只生效 3 次 | 每个线程读到的值相同，后来的 UPDATE 覆盖了前面的 |

MySQL 的行锁只在 `UPDATE` 执行时生效，但 `SELECT` 在 `UPDATE` 之外，无法阻止其他线程先读到旧值。

### V0 知识点（面试用）

| 概念 | 说明 | 在本项目中的体现 |
|---|---|---|
| **临界区** | 多个线程并发访问的共享资源代码块 | `SELECT stock → if>0 → UPDATE stock` 这段代码 |
| **竞态条件** | 多线程执行结果取决于线程的调度顺序 | 两个线程都读到 20，都减到 19，最终结果取决于谁最后写 |
| **原子性** | 一个操作不可被中断，要么全部执行要么全部不执行 | `selectById` 和 `updateById` 是两个独立操作，整体不原子 |
| **丢失更新** | 一个线程的写入被另一个线程的写入覆盖 | A 写 stock=19，B 也写 stock=19，A 的更新丢失了 |

## V1 — synchronized 关键字

> **解决目标**：保证临界区代码同一时刻只有一个线程执行

### 解决思路

在 `StockService.decrease()` 方法上加 `synchronized`，让多个线程串行访问库存扣减逻辑：

```java
// V1 修改：在方法上直接加 synchronized
public synchronized Result<String> decrease(Long goodsId) {
    GoodsStock stock = stockMapper.selectById(goodsId);
    if (stock.getStock() <= 0) {
        return Result.fail("库存不足");
    }
    stock.setStock(stock.getStock() - 1);
    stockMapper.updateById(stock);
    return Result.success("ok");
}
```

### 改造范围

| 文件 | 修改内容 |
|---|---|
| `seckill-stock/.../StockService.java` | `decrease()` 方法加 `synchronized` 关键字 |
| `seckill-payment/.../PaymentService.java` | `pay()` 方法加 `synchronized` 关键字 |

### 效果

50 并发压测后：

```
初始库存: 20
成功订单: 20 笔      ← 刚好等于库存，没有超卖
剩余库存: 0          ← 全部扣完
失败订单: 30 笔      ← "库存不足"
```

`synchronized` 保证 `SELECT → if → UPDATE` 同一时刻只有一个线程执行。线程 B 必须等线程 A 全部执行完（包括 UPDATE 提交）才能进入。

### V1 的局限（面试重点）

| 局限 | 说明 |
|---|---|
| **仅限单机** | `synchronized` 是 JVM 级别的锁。两个服务实例就是两把不同的锁，分布式下仍然超卖 |
| **性能下降** | 串行化处理，QPS 受限于单线程吞吐，无法水平扩展 |
| **锁范围过大** | `public synchronized` 锁的是 this 对象，所有商品共用一个锁。如果有 A/B 两个商品，买 A 的线程会阻塞买 B 的线程 |

### synchronized 知识点（面试用）

#### 1. 基本用法

```java
// 用法一：同步方法 —— 锁的是当前对象（this）
public synchronized Result<String> decrease() { ... }

// 用法二：同步代码块 —— 细粒度控制锁对象
public Result<String> decrease() {
    synchronized (this) {
        // 只锁这里
    }
}

// 用法三：静态方法 —— 锁的是 Class 对象
public static synchronized void reset() { ... }
```

#### 2. 锁的是什么？

| 用法 | 锁的对象 |
|---|---|
| `synchronized` 实例方法 | 当前对象 `this` |
| `synchronized` 静态方法 | 当前类 `Class` 对象 |
| `synchronized(obj)` | 指定的 `obj` 对象 |

#### 3. 锁的升级过程（JDK 1.6 优化）

`synchronized` 早期是重量级锁（依赖操作系统 Mutex），JDK 1.6 引入锁升级机制：

```
无锁 → 偏向锁 → 轻量级锁（自旋锁）→ 重量级锁
```

| 阶段 | 触发条件 | 特点 |
|---|---|---|
| **偏向锁** | 同一个线程反复获取锁 | 只在对象头做标记，不阻塞 |
| **轻量级锁** | 少量线程交替获取锁 | 自旋等待，不挂起线程 |
| **重量级锁** | 多个线程同时竞争锁 | 线程挂起，依赖 OS 调度 |

#### 4. 与 volatile 的区别

| | synchronized | volatile |
|---|---|---|
| 原子性 | ✅ 保证 | ❌ 不保证 |
| 可见性 | ✅ 保证 | ✅ 保证 |
| 有序性 | ✅ 保证 | ✅ 禁止指令重排 |
| 使用场景 | 复合操作（读-改-写）| 单一变量的可见性 |

#### 5. 常见面试题

**Q：synchronized 和 Lock 的区别？**
- synchronized 是 JVM 内置关键字，自动释放锁；Lock 是 API 层面，需要手动 `unlock()`
- synchronized 不可中断；Lock 可中断（`lockInterruptibly()`）
- synchronized 只有非公平锁；Lock 可设置公平/非公平
- synchronized 发生异常自动释放锁；Lock 需要在 finally 中释放

**Q：为什么 V1 的 synchronized 在微服务架构下没用？**
- 两个服务实例各有一个 JVM，各有一把锁
- 线程 A 拿到服务 1 的锁，线程 B 拿到服务 2 的锁，两把锁互不相干
- 分布式场景需要**分布式锁**（V3 引入 Redis 锁）
