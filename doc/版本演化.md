## V0 — 无锁直扣 MySQL（演示超卖）

### 版本说明

最原始的实现。不做任何并发控制，直接 `SELECT → 判断 → UPDATE` 操作数据库。

### 代码位置

| 文件 | 关键代码 |
|---|---|
| `StockService.java` | `selectById(goodsId)` → `if(stock>0)` → `setStock(stock-1)` → `updateById(stock)` |
| `PaymentService.java` | `selectOne(userId)` → `if(balance>amount)` → `setBalance(balance-amount)` → `updateById` |
| `SeckillController.java` | 三次 Feign 调用：stock → payment → order，无事务 |

### 复现方式

```sql
-- 库存设 20，50 并发压测
UPDATE goods_stock SET stock = 20 WHERE id = 1;
```

Apifox 并发测试：50 线程同时请求 `POST /api/seckill/buy?userId=10001&goodsId=1`

### 观察到的现象

```
初始库存: 20
成功订单: 25 笔      ← 超卖了！
剩余库存: 17         ← 只有 3 次更新真正生效
```

### 根因分析

**非原子的"读-改-写"三步骤**：

```java
// StockService.java 第21~38行
GoodsStock stock = stockMapper.selectById(goodsId);    // Step1: 读
if (stock.getStock() <= 0) { return fail(...); }        // Step2: 判断
stock.setStock(stock.getStock() - 1);                   // Step3: 减
stockMapper.updateById(stock);                          // Step4: 写
```

高并发下多个线程**同时**读到了相同的库存值：

```
时间线   线程A                    线程B                    DB stock
 │       SELECT stock=20                               20
 │                             SELECT stock=20         20
 │       if(20>0)=true         if(20>0)=true
 │       UPDATE stock=19                               19
 │                             UPDATE stock=19         19 ← B覆盖了A！
 │       return success        return success
 │       创建订单#1             创建订单#2
```

**两个问题叠加：**

| 问题 | 表现 | 原因 |
|---|---|---|
| **超卖** | 20 库存卖出 25 单 | 多个线程同时通过 `if(stock>0)` 检查 |
| **更新丢失** | 25 次扣减只生效 3 次 | 每个线程读到的值相同，后来的 UPDATE 覆盖了前面的 |

MySQL 的行锁只在 `UPDATE` 执行时生效，但 `SELECT` 在 `UPDATE` 之外，无法阻止其他线程先读到旧值。

### V0 知识点（面试用）

| 概念 | 说明 | 在本项目中的体现 |
|---|---|---|
| **临界区** | 多个线程并发访问的共享资源代码块 | `SELECT stock → if>0 → UPDATE stock` 这段代码 |
| **竞态条件** | 多线程执行结果取决于线程的调度顺序 | 两个线程都读到 20，都减到 19，最终结果取决于谁最后写 |
| **原子性** | 一个操作不可被中断，要么全部执行要么全部不执行 | `selectById` 和 `updateById` 是两个独立操作，整体不原子 |
| **丢失更新** | 一个线程的写入被另一个线程的写入覆盖 | A 写 stock=19，B 也写 stock=19，A 的更新丢失了 |

## V1 — synchronized 关键字

> **解决目标**：保证临界区代码同一时刻只有一个线程执行

### 解决思路

在 `StockService.decrease()` 方法上加 `synchronized`，让多个线程串行访问库存扣减逻辑：

```java
// V1 修改：在方法上直接加 synchronized
public synchronized Result<String> decrease(Long goodsId) {
    GoodsStock stock = stockMapper.selectById(goodsId);
    if (stock.getStock() <= 0) {
        return Result.fail("库存不足");
    }
    stock.setStock(stock.getStock() - 1);
    stockMapper.updateById(stock);
    return Result.success("ok");
}
```

### 改造范围

| 文件 | 修改内容 |
|---|---|
| `seckill-stock/.../StockService.java` | `decrease()` 方法加 `synchronized` 关键字 |
| `seckill-payment/.../PaymentService.java` | `pay()` 方法加 `synchronized` 关键字 |

### 效果

50 并发压测后：

```
初始库存: 20
成功订单: 20 笔      ← 刚好等于库存，没有超卖
剩余库存: 0          ← 全部扣完
失败订单: 30 笔      ← "库存不足"
```

`synchronized` 保证 `SELECT → if → UPDATE` 同一时刻只有一个线程执行。线程 B 必须等线程 A 全部执行完（包括 UPDATE 提交）才能进入。

### V1 的局限（面试重点）

| 局限 | 说明 |
|---|---|
| **仅限单机** | `synchronized` 是 JVM 级别的锁。两个服务实例就是两把不同的锁，分布式下仍然超卖 |
| **性能下降** | 串行化处理，QPS 受限于单线程吞吐，无法水平扩展 |
| **锁范围过大** | `public synchronized` 锁的是 this 对象，所有商品共用一个锁。如果有 A/B 两个商品，买 A 的线程会阻塞买 B 的线程 |

#### 多实例场景下的失效分析

**测试验证：**

```
部署 2 个 seckill-stock 实例（8082, 8083）
库存: 20
并发: 50

结果：
- 50 次请求全部返回 HTTP 200
- 库存扣完（归零）
- 但实际订单只有 12 笔
```

**失效原因：** 两个 JVM 各有一把 `synchronized` 锁。线程 A 拿到实例 8082 的锁，线程 B 同时拿到实例 8083 的锁，两把锁互不相干。两个线程同时读到 `stock=20`，都更新为 `19`，后一个覆盖前一个 —— 更新丢失。

#### 标准解决方案：分布式锁 + synchronized

在生产环境中，通常**两层锁配合**使用：

```java
public Result<String> decrease(Long goodsId) {
    String lockKey = "lock:stock:" + goodsId;
    String lockValue = UUID.randomUUID().toString();
    // 第一层：Redis 分布式锁（跨 JVM）
    Boolean locked = redisTemplate.opsForValue()
        .setIfAbsent(lockKey, lockValue, 10, TimeUnit.SECONDS);
    if (!locked) {
        return Result.fail("请求太频繁");
    }
    try {
        // 第二层：synchronized（单 JVM 内兜底，防止同实例内线程竞争）
        synchronized (this) {
            GoodsStock stock = stockMapper.selectById(goodsId);
            if (stock.getStock() <= 0) return Result.fail("库存不足");
            stock.setStock(stock.getStock() - 1);
            stockMapper.updateById(stock);
            return Result.success("ok");
        }
    } finally {
        unlock(lockKey, lockValue);
    }
}
```

| 层 | 作用 | 解决场景 |
|---|---|---|
| **Redis 分布式锁** | 跨 JVM 互斥 | 实例 8082 和 8083 的线程同时操作 |
| **synchronized** | 单 JVM 兜底 | 同一实例内多个线程同时竞争 |

两层锁覆盖了"单实例 + 多实例"两种场景，确保任何部署方式下都不会超卖。

### synchronized 知识点（面试用）

#### 1. 基本用法

```java
// 用法一：同步方法 —— 锁的是当前对象（this）
public synchronized Result<String> decrease() { ... }

// 用法二：同步代码块 —— 细粒度控制锁对象
public Result<String> decrease() {
    synchronized (this) {
        // 只锁这里
    }
}

// 用法三：静态方法 —— 锁的是 Class 对象
public static synchronized void reset() { ... }
```

#### 2. 锁的是什么？

| 用法 | 锁的对象 |
|---|---|
| `synchronized` 实例方法 | 当前对象 `this` |
| `synchronized` 静态方法 | 当前类 `Class` 对象 |
| `synchronized(obj)` | 指定的 `obj` 对象 |

#### 3. 锁的升级过程（JDK 1.6 优化）

##### 为什么需要锁升级

早期 JDK 的 `synchronized` 直接使用**重量级锁**（依赖操作系统 Mutex / Monitor），每次加锁解锁都需要在用户态和内核态之间切换，上下文切换的开销非常大。而很多并发场景根本用不上这么重的锁。

所以 JDK 1.6 引入了锁升级机制，让 `synchronized` 能根据竞争激烈程度自动适配：**竞争少时轻量处理，竞争激烈时升级为重量级锁**。

##### 升级路径

```
偏向锁 → 轻量级锁（CAS 自旋）→ 重量级锁（OS Mutex）
```

| 阶段 | 触发条件 | 原理 | 特点 |
|---|---|---|---|
| **偏向锁** | 只有一个线程反复获取锁 | 在对象头的 Mark Word 中记录线程 ID，每次进入时检查 ID 是否一致，一致则不操作 | 本质上是单线程，零开销 |
| **轻量级锁** | 多个线程交替获取锁，但**不同时竞争** | 使用 CAS 自旋尝试替换对象头的锁记录。自旋是忙等（空转 CPU），不阻塞线程 | 延迟低，适合短时间锁竞争 |
| **重量级锁** | 多个线程**同时竞争**锁，CAS 自旋失败或达到自旋阈值 | 未获取到锁的线程进入阻塞状态（`BLOCKED` / `WAITING`），不再消耗 CPU。锁释放时由 OS 唤醒等待线程 | 适合长时间锁竞争，避免 CPU 空转 |

##### 关键细节

- **单向升级**：锁只能升级，不能降级。即使竞争不再激烈，重量级锁也不会退回轻量级锁
- **自旋阈值**：JDK 1.6 默认自旋 10 次，超过则升级。`-XX:PreBlockSpin` 可配置
- **自适应自旋**：JDK 1.6 引入，JVM 根据上次自旋成功率自动调整自旋次数

##### 实际场景的表现

```
线程交替获取锁（无竞争） ──→ 偏向锁 / 轻量级锁（CAS 自旋，不阻塞）
                                    ↓
多个线程同时竞争锁（高并发） ──→ 重量级锁（线程阻塞，CPU 让出）
```

一般场景下锁都停留在**轻量级锁**阶段，只有高并发激烈竞争时才升级到重量级锁。这就是为什么 `synchronized` 在 JDK 1.6 之后性能大幅提升——大部分场景下它根本不进内核态。

#### 4. 与 volatile 的区别

| | synchronized | volatile |
|---|---|---|
| 原子性 | ✅ 保证 | ❌ 不保证 |
| 可见性 | ✅ 保证 | ✅ 保证 |
| 有序性 | ✅ 保证 | ✅ 禁止指令重排 |
| 使用场景 | 复合操作（读-改-写）| 单一变量的可见性 |

#### 5. 常见面试题

**Q：synchronized 和 Lock 的区别？**
- synchronized 是 JVM 内置关键字，自动释放锁；Lock 是 API 层面，需要手动 `unlock()`
- synchronized 不可中断；Lock 可中断（`lockInterruptibly()`）
- synchronized 只有非公平锁；Lock 可设置公平/非公平
- synchronized 发生异常自动释放锁；Lock 需要在 finally 中释放

**Q：为什么 V1 的 synchronized 在微服务架构下没用？**
- 两个服务实例各有一个 JVM，各有一把锁
- 线程 A 拿到服务 1 的锁，线程 B 拿到服务 2 的锁，两把锁互不相干
- 分布式场景需要**分布式锁**（V3 引入 Redis 锁）

## 分布式锁与事务提交时序问题

> 面试高频：synchronized + @Transactional + Redis 分布式锁 的组合陷阱

### 现象描述

```java
// 错误示例
@Service
public class OrderService {

    @Transactional                    // 事务在 AOP 后置处理中提交
    public synchronized void A() {    // synchronized 在方法结束后释放
        // 业务逻辑...
        try {
            // ...
        } finally {
            redisLock.unlock();       // Redis 锁在 finally 中释放，比 synchronized 还早
        }
    }
}
```

三个锁的释放时序：

```
方法开始 → 获取 Redis 锁 → 执行业务 → finally 释放 Redis 锁 → 方法返回 → synchronized 释放 → AOP 提交事务
                            │                                  │                │            │
                            │                              Redis 锁提前释放       │            │
                            │                                                  synchronized 释放 │
                            │                                                                事务提交
```

### 问题所在

| 锁 | 释放时机 | 释放时事务状态 |
|---|---|---|
| **Redis 锁** | `finally` 块中 | 事务还未提交 |
| **synchronized** | 方法执行完毕时 | 事务还未提交 |
| **@Transactional** | AOP 后置通知中 `commit()` | 事务提交 |

**事务还没提交，锁就释放了** —— 其他线程/服务看到锁没了，进入临界区，读取到的是事务**未提交**的旧数据（脏读/不可重复读），产生并发异常。

### 正确做法

#### 方案一：锁外事务内（推荐）

```java
@Service
public class OrderService {

    // A：加锁，不处理事务
    public synchronized void A() {
        B();  // 调用事务方法
    }

    // B：处理事务，不加锁
    @Transactional
    public void B() {
        // 业务逻辑
    }
}
```

**锁的时序：**

```
方法 A 开始 → synchronized 获取锁 → 调用 B → B 执行事务 → B 返回 → A 结束 → synchronized 释放锁
                                     │                                  │
                                     │ 事务在 B 方法结束时提交            │
                                     │                                  │
                                     │                      事务已提交，再释放锁
```

**关键点：A 和 B 不能在同一个类中**，否则 `@Transactional` 的 AOP 代理不会生效（同类调用不走代理）。

#### 方案二：Redis 分布式锁 + 事务分离

```java
@Service
public class SeckillService {

    // C：获取 Redis 锁，再调用业务方法
    public Result<String> C(Long goodsId) {
        String lockKey = "lock:seckill:" + goodsId;
        String lockValue = UUID.randomUUID().toString();
        try {
            // 获取 Redis 锁
            Boolean locked = redisTemplate.opsForValue()
                .setIfAbsent(lockKey, lockValue, 10, TimeUnit.SECONDS);
            if (!locked) {
                return Result.fail("请求太频繁");
            }
            // 锁内调用事务方法
            return A(goodsId);
        } finally {
            // 事务提交后再释放锁
            unlock(lockKey, lockValue);
        }
    }

    // A：业务逻辑 + synchronized（可选，作为兜底）
    public synchronized Result<String> A(Long goodsId) {
        return B(goodsId);  // 调用另一个类的事务方法
    }
}

// 另一个类
@Service
public class StockDbService {
    @Transactional
    public Result<String> B(Long goodsId) {
        // 数据库操作
    }
}
```

**锁的时序：**

```
C 开始 → 获取 Redis 锁 → 调用 A → A 获取 synchronized → 调用 B → B 提交事务 → B 返回 → A 释放 synchronized → C 的 finally 释放 Redis 锁
                                                                    │                                            │
                                                                    事务已提交                                     事务早已提交
```

此时即使其他线程/服务看到锁释放后进入，读到的一定是已提交的数据。

### 核心原则

| 原则 | 说明 |
|---|---|
| **锁范围 ≥ 事务范围** | 锁释放必须在事务提交**之后**，否则锁失去意义 |
| **锁内调用事务** | 加锁的方法调用事务方法，而不是事务方法内加锁 |
| **跨类调用** | Spring AOP 代理只在跨类调用时生效，同类调用 `@Transactional` 无效 |
| **Redis 锁不在 finally 提前放** | 确保事务提交后再释放分布式锁 |

### 面试追问

**Q：为什么 synchronized 和 @Transactional 不能加在同一个方法上？**
- synchronized 是 Java 关键字，作用于方法字节码层面
- @Transactional 是 Spring AOP，通过代理在方法前后插入 `begin()` / `commit()`
- AOP 的后置 `commit()` 在方法返回后执行，而 synchronized 在方法返回时释放
- 结果：锁先释放，事务后提交，数据不一致

**Q：怎么保证 Redis 锁在事务提交后才释放？**
- 把事务方法拆到另一个类中，锁方法调用事务方法
- 锁方法等事务方法返回后（事务已提交）再释放 Redis 锁
- 代码结构：`获取锁 → 调用事务方法（提交事务）→ 释放锁`

V2：实现ReentrantLock调用链路，ReentrantLock是在类的内部实现了一个内部类Sync继承了AQS，然后实现了公平锁和非公平锁，支持tryLock，可以自己手动上锁和解锁，更加灵活。
